Your Home Server: The Learning Edition

Build a real home server and, without really trying, learn the foundations of modern IT: virtualization, Linux, networking, containers and security. Every step is also a lesson, so you finish understanding what you did and why.

How this edition works It still walks you through every click, but now each step is a lesson. Watch for these along the way: a teal "Understand this" box goes one level deeper into how something really works; an indigo "You'll meet this again" box connects what you just did to AZ-900, the cloud, and the wider industry; each section ends with a green recap and a couple of check-yourself questions. Read the boxes, attempt the questions before reading the answers, and you'll come out the other side genuinely understanding this stuff, not just having followed it.
Why this is a shortcut into tech Courses like AZ-900 describe things like "virtual machines", "networking" and "containers" from a height. The reason they can feel slippery is that you've never held them. This project makes you build all of them by hand on your own box. When AZ-900 later says "Azure runs your VM on a hypervisor", you'll think "right, like Proxmox on my OptiPlex" instead of memorising a definition. That is the difference between passing and understanding, and it's also how you find out which parts of tech you actually enjoy.

2.The mental model everything hangs off

Five ideas. Read once; the rest of the guide keeps pointing back to them.

1 · Home network vs the internet Everything on your wifi is your local network (the inside). The internet is the outside. Your router is the guarded door between them, and by default it lets conversations out but blocks uninvited ones coming in. Your server lives inside.
2 · An IP address is a house number Every device has an IP address like 192.168.1.42 so things can find each other. Addresses starting 192.168. (also 10. and 172.16-31.) are private: they only mean something inside your network. The outside world can't see them.
3 · DNS is the internet's phone book Type bbc.co.uk and your device asks a DNS service "what's the number for that name?" and gets an IP back. Pi-hole will be your own phone book that answers normally but refuses to look up ad/tracker names, so ads never load.
4 · A hypervisor makes one computer act like many Proxmox is a hypervisor: software whose whole job is to run several separate virtual computers on one physical box, each walled off from the others. Each little one here is a container.
5 · A snapshot is a save point Proxmox can photograph a container's exact state before you change it. Break something, "roll back", and it's as it was in seconds. This is your safety net and you'll lean on it constantly.
You'll meet this again Ideas 1 to 3 are the core of the CompTIA Network+ and the networking slice of every cloud cert. Idea 4 (virtualization) is the foundation of IaaS in AZ-900: an Azure VM is just a virtual machine on Microsoft's hypervisors instead of yours. Idea 5 is the seed of backup & disaster recovery, a topic in its own right.
Recap
  • Inside (local network) and outside (internet) are separated by the router on purpose.
  • Private IPs identify devices inside; DNS turns names into those numbers.
  • A hypervisor runs many virtual machines/containers on one physical machine; snapshots let you undo.
Check yourself

Q1. Your friend says "just put your Jellyfin on 192.168.1.60 and I'll watch it from my house." Why won't that work as-is?

192.168.x is a private address that only exists inside your own network, so it means nothing from their house. Reaching it from outside needs something extra (later: Tailscale), never just the private IP.

Q2. In one sentence, what does a hypervisor do?

It runs multiple isolated virtual computers on a single physical machine.

3.Prep, and the vocabulary to have ready

  • The OptiPlex, RAM and SSD fitted, plugged into power and into your router with an ethernet cable. wired = simpler + steadier
  • A USB stick (2GB+, gets wiped).
  • Your laptop on the same wifi, for the browser side.
  • Your router's admin login (often on a sticker on the router). Find it now.
  • A notes file for addresses and passwords as you make them. Treat this as part of the skill: professionals document everything.
Understand this: why wired matters An always-on server on wifi has to keep re-negotiating a radio link that drops and shares airtime with everything else. A wired ethernet connection is a dedicated, stable path. In a data centre or the cloud, servers are never on wifi for exactly this reason. Small detail, real-world principle.
Part 2 · Build the machine and its core services

4.Installing Proxmox: meet the hypervisor

Understand this before you start: what is Proxmox? Proxmox VE is a free operating system built on Debian Linux whose job is to be a Type 1 hypervisor. "Type 1" means it installs directly onto the bare hardware and the virtual machines run on top of it (as opposed to "Type 2" like VirtualBox, which runs as an app inside Windows). Type 1 is what real servers and clouds use because there's no extra operating system in the way stealing performance. You're installing the same class of software that runs Azure, AWS and Google Cloud, just the home edition.

Step A, make the install USB you do this by hand

  1. On your laptop, go to proxmox.com → Downloads → Proxmox VE, and download the .iso file.
    What an ISO is: a single file that is an exact image of an install disc. It contains the whole operating system, ready to be written to a USB and booted.
  2. Download balenaEtcher from balena.io/etcher, open it, choose Flash from file → the ISO, Select target → your USB stick (check it's the stick!), then Flash.
    Why not just copy it on: a bootable stick needs the data written in a special low-level layout the computer can start from, which Etcher does. Copy-paste wouldn't boot.
    You'll see: "Flash Complete." The stick is now an installer.

Step B, install onto the OptiPlex by hand

  1. Insert the stick, power on, tap F12 repeatedly, and pick the USB from the boot menu.
    What's happening: the "boot menu" lets you choose which drive the machine starts from this once. Normally it starts from its internal drive; you're telling it to start from your installer instead.
  2. Choose Install Proxmox VE (Graphical) and accept the licence.
  3. Target drive: choose your Crucial SSD (check the size looks right, ~480GB).
    Understand this: the installer will format this drive and lay down a Linux filesystem plus Proxmox. "Format" means wipe and set up a fresh filing system. It's the one destructive moment, hence the check.
  4. Country / timezone / password / email: set a strong root password (write it down) and any email.
    Who "root" is: on Linux, root is the all-powerful administrator account (the equivalent of full admin on Windows). This password guards total control of the machine, so it matters.
  5. Network screen: note the IP Address it shows (e.g. 192.168.1.50). Accept sensible defaults for the rest.
    Why write it down: that address is how you'll reach the server from your laptop. It's the front-door number for everything from here on.
  6. Click Install, let it finish, and remove the USB as it reboots.
    You'll see: a blue screen ending with something like https://192.168.1.50:8006. That's your control panel address. The machine can now run "headless" (no monitor or keyboard).
Understand this: "headless" and why servers are like it A headless machine has no screen, keyboard or mouse; you administer it entirely over the network from another device. Nearly all servers on Earth are headless, sitting in racks nobody looks at, managed remotely. You just set one up. That web address on port :8006 is your remote management console.
You'll meet this again This is IaaS (Infrastructure as a Service) from the ground up, an explicit AZ-900 topic. When Azure gives you a VM, they've done what you just did (hypervisor on hardware) and handed you the "create a virtual machine" button. You now know what's behind that button.
Recap
  • Proxmox is a Type 1 (bare-metal) hypervisor built on Debian Linux.
  • An ISO is a bootable disc image; you flashed it to USB to install from.
  • root is the Linux super-admin; servers run headless, managed over the network on a port.
Check yourself

Q1. What's the difference between a Type 1 and Type 2 hypervisor?

Type 1 runs directly on the hardware (Proxmox, ESXi, Azure's); Type 2 runs as an application inside an existing OS (VirtualBox). Type 1 is faster and used for servers/cloud.

Q2. You set a password for an account called "root". What is root?

The all-powerful administrator account on a Linux system, with complete control over the machine.

5.First login: Linux, packages and the command line

  1. On your laptop browser, go to your https://192.168.1.50:8006 address (include https:// and :8006).
    What the :8006 means: a single machine can run many services at once, so each listens on a numbered port, like flat numbers at one street address. Proxmox's web console lives on port 8006. You'll see this idea everywhere.
  2. The browser warns "not private". Here that's expected: click Advanced → Proceed.
    Understand this: that warning means the site's security certificate isn't signed by a public authority. For a public bank site that'd be alarming; for your own machine on your own network it's normal, because no public authority has vouched for your private box. The connection is still encrypted.
  3. Log in with root and your password. Dismiss the "No valid subscription" notice (it's an optional paid support plan).
    You'll see: the Proxmox dashboard. On the left is your server ("node"); clicking it reveals tabs.
  4. Click your node, then Shell. A black terminal opens. Type hostnamectl and press Enter.
    What you just did: the Shell is a command-line terminal on the server itself. hostnamectl prints system info. You're now "in" a Linux machine, typing commands, the way most of the server world is actually run.
Understand this: how Linux installs software (this is a big one) On Windows you download an .exe and double-click. Linux mostly uses a package manager: a built-in tool that fetches software from trusted online repositories (curated software libraries) and installs it with dependencies handled for you. Proxmox is Debian-based, so its package manager is apt. Two commands do almost everything: apt update # refresh the list of available software apt upgrade # install newer versions of what you have This is one of the most useful things you can learn, because every Debian/Ubuntu server on the planet works this way.

The one setup script, and what it's actually doing script for speed, explained fully

  1. Open community-scripts.org, find Proxmox VE Post Install, copy the command into the Shell, run it, accept the sensible defaults, and reboot when asked.
    What it does under the hood, so you learn it: a fresh Proxmox is configured to use the enterprise software repository, which needs a paid subscription, so updates fail and it nags you. The script switches you to the free no-subscription repository and turns off the nag. In plain apt terms it's editing the repository list (files under /etc/apt/) and running apt update. You could do it by hand; the script just saves fiddly typing.
    You'll see: text scrolling as it edits repositories and refreshes, then a reboot prompt.
The rule for the Shell (a real security habit) Only paste commands from an official project's own site (community-scripts.org, or a vendor's own domain like tailscale.com later). Pasting a command you don't understand from a random forum is exactly how people get compromised. "Don't run untrusted code as root" is a principle you'll see stated formally in every security course.
You'll meet this again Package managers, repositories and the command line are day-one Linux skills. They're the core of CompTIA Linux+ and assumed knowledge for AZ-104, DevOps, and cloud engineering (cloud servers are overwhelmingly Linux). The comfort you build typing in this Shell is directly transferable and genuinely uncommon at entry level, so it stands out.
Recap
  • Services listen on numbered ports (Proxmox = 8006); one machine, many services.
  • The Shell is a Linux command line on the server; you ran your first commands.
  • Linux installs software via a package manager (apt) from repositories; the post-install script just automated a repo change.
Check yourself

Q1. What does apt update do, and how does it differ from apt upgrade?

apt update refreshes the list of what software is available; apt upgrade actually installs the newer versions. Update = check, upgrade = apply.

Q2. Why does the browser warn that Proxmox "isn't private", and why is it fine here?

Its security certificate isn't signed by a public authority (it's self-signed for a private machine). It's fine because it's your own server on your own network; the connection is still encrypted.

6.Your first container + Pi-hole: containers & networking

This is the richest learning section, so we build the container by hand (it's the single best way to understand virtualization), then install Pi-hole inside it and learn how networks actually resolve names.

Understand this: container vs virtual machine Both give you an isolated "computer", but differently. A virtual machine pretends to be a whole physical PC and runs its own complete operating system, so it's heavier. A container shares the host's Linux core (kernel) and just isolates the software on top, so it's much lighter and starts in seconds. Proxmox's containers use LXC. Rule of thumb: containers for lightweight Linux services (like Pi-hole), VMs when you need a full separate OS (like Windows). This exact trade-off is a huge topic in industry, where Docker and Kubernetes made containers the default way to ship software.

Step A, create your first container by hand you do this by hand

  1. In Proxmox, first get a "template" (a ready-made mini OS to base the container on): click your node → local (storage) → CT Templates → Templates, and download debian-12-standard.
    What a template is: a pre-built base image of a minimal operating system. Your container starts as a copy of it, so you're not installing an OS from scratch each time. Cloud works the same way ("images").
  2. Top-right, click Create CT (CT = container). Give it a hostname like pihole, and set a password for inside the container.
  3. Choose the debian-12 template you downloaded. For resources, modest is fine: 1 CPU core, 512MB memory, 4GB disk.
    Understand this: you're carving off a slice of the physical machine's cores, memory and disk to give this little computer. This is exactly what "sizing a VM" means in the cloud, where you pick how much CPU/RAM to rent. Pi-hole is tiny, so it needs almost nothing.
  4. On the network screen, leave it to get its address automatically (DHCP) for now, finish the wizard, and Start the container.
    What DHCP is: "Dynamic Host Configuration Protocol", the service (usually on your router) that automatically hands out IP addresses to devices that join the network, so you don't set them by hand. Your phone, laptop and now this container all get theirs this way.
    You'll see: your container listed in the left panel, and under its Summary the IP it was given, e.g. 192.168.1.51. You built a working computer in about two minutes.
Pin its address (so networking stays predictable) Because the whole house will soon rely on Pi-hole's address, it shouldn't change. Easiest way: in your router, find DHCP reservation and tie Pi-hole's address to it permanently. Why this matters technically: DHCP leases can change over time; a reservation (or a manually set "static" IP) guarantees a fixed number. Servers almost always get fixed addresses for this reason.

Step B, install Pi-hole inside the container

  1. Select the container → Console, log in as root with the password you set, and run Pi-hole's official installer: curl -sSL https://install.pi-hole.net | bash
    What this line means, piece by piece: curl downloads a file from a web address; the | ("pipe") feeds that downloaded script straight into bash, which runs it. It's from Pi-hole's official domain, so it's trusted. Follow the prompts, accepting defaults; when it offers an upstream DNS provider pick one like Cloudflare.
    You'll see: a blue setup wizard, then a final screen giving you the admin web address (http://192.168.1.51/admin) and a password. Write both down.
Understand this: what actually happens when you open a website This is the concept Pi-hole teaches, and it's gold for interviews and certs. When you type bbc.co.uk: 
your device ──"what's the IP for bbc.co.uk?"──▶ Pi-hole (your DNS resolver)
      │                                              │
      │                          is it on the ad/tracker blocklist?
      │                          ├─ yes ▶ reply "nothing here" (ad never loads)
      │                          └─ no  ▶ ask an upstream DNS (e.g. Cloudflare)
      │                                     which returns the real IP
      ◀──────────── real IP (e.g. 151.101.0.81) ─────────────┘
your device then connects to that IP to load the page
DNS runs on port 53. Pi-hole is simply a DNS resolver that consults a blocklist first. That's the entire trick, and now you understand it properly.

Step C, roll it out safely (test on one device first)

This is the one change that touches the whole household, so we de-risk it exactly like a professional would: test on a single device, keep the undo ready, then roll out.

  1. Test on your phone only: phone wifi settings → your network → DNS → switch to Manual → enter Pi-hole's IP (192.168.1.51). Browse.
    You'll see: sites load, many ads vanish, and Pi-hole's dashboard counter climbs. Proof, affecting nobody else.
  2. Then roll out to everyone: log into your router, find the DNS server setting, note the current value, and replace it with Pi-hole's IP. Save, then reconnect a device.
    Why the router: it hands out DNS settings to every device via DHCP, so changing it once points the whole house at Pi-hole automatically.
If it goes wrong (know this before you touch the router) If the house can't load sites, set the router's DNS back to Automatic / the value you noted. Everyone is back to normal within a minute. This never touches your actual internet line, only which phone book devices consult, so it's fully reversible. Also good to know: if you later shut the server off for a while, switch the router DNS back first, or the house will struggle to look names up until it returns.
You'll meet this again DNS, DHCP, ports and IP addressing are the backbone of CompTIA Network+ and appear in every cloud platform: Azure has Azure DNS and hands out private IPs to VMs inside virtual networks exactly like your router does at home. "It's always DNS" is the industry's favourite troubleshooting joke because DNS problems are so common; you now know why.
Recap
  • Containers are lightweight isolated computers sharing the host kernel; you built one from a template and sized its CPU/RAM/disk.
  • DHCP auto-assigns IPs; a reservation pins one so servers stay findable.
  • DNS resolves names to IPs on port 53; Pi-hole is a resolver that filters a blocklist first.
  • You rolled out a network-wide change the safe way: test one device, keep the undo, then apply.
Check yourself

Q1. A container and a VM both isolate a workload. What's the key technical difference?

A VM runs its own full operating system (heavier); a container shares the host's kernel and only isolates the software on top (lighter, faster to start).

Q2. Walk through what DNS does when you visit a site, in one sentence.

Your device asks a DNS resolver for the IP address behind the name, and once it gets the number it connects to that IP to load the page.

Q3. Why give the Pi-hole container a reserved (fixed) IP?

Because the whole network will point at that address for DNS; if DHCP changed it, lookups would break. Servers need predictable addresses.

7.Storage: giving Jellyfin somewhere to live

Your SSD runs the system; this new hard drive holds the media. Before Jellyfin can use it you have to prepare the drive, attach it into Linux, and open a controlled doorway from the host into the Jellyfin container. Do this once Proxmox is running and after the Jellyfin container exists, so you have something to point it at.

Understand this: SSD for the system, hard drive for the data Two jobs, two kinds of drive. Your SSD is fast but small: ideal for the operating system and apps that do constant small reads and writes. Your NAS hard drive is big and cheap per terabyte: ideal for bulk media that's read in long steady streams and doesn't need speed. Splitting "fast small disk for the system" from "big cheap disk for the data" is a standard design you'll see everywhere, including cloud VMs, which pair a small OS disk with larger attached data disks for exactly this reason.

Step A, fit the drive by hand

  1. With the machine off, connect the hard drive's SATA data and SATA power cables (same as the SSD) and secure it. Power back on.
    Why alongside, not instead: the SSD stays as your system drive; this is a second, separate drive purely for files.

Step B, put a filesystem on it by hand, strong learning

Understand this: raw disk vs filesystem A brand-new drive is just raw empty space; the system can't store files on it until you put a filesystem on it. A filesystem is the organising structure (folders, names, where each file physically lives) that turns raw space into something usable, like fitting an empty warehouse with labelled shelving. "Formatting" means laying that structure down. Windows uses one called NTFS; Linux most commonly uses ext4, which is what you'll use.
  1. In the Proxmox Shell, list the drives to find the new one: lsblk
    Reading it: this lists every disk with its size. Your SSD (~480G, with Proxmox on it) will be there; the empty ~8T disk is the new one, most likely called /dev/sdb. Note its name carefully: the next command wipes whatever you name.
  2. Format the new drive with ext4 (swap sdb for whatever lsblk showed): mkfs.ext4 /dev/sdb
    What this does: writes the ext4 filesystem onto the drive, making it ready to hold files. It erases the drive, which is fine because it's new and empty. This is exactly why you double-checked the name first.

Step C, mount it into Linux

Understand this: mounting, and Linux's single tree Windows gives each drive its own letter (C:, D:). Linux is different, and once it clicks, cleaner: there is one folder tree starting at / (the "root"), and you mount a drive into that tree at a folder of your choosing. Your media drive won't be "D:"; it'll appear as, say, /mnt/media. Mounting is simply attaching a drive to a chosen spot in the tree.
  1. Make a folder to mount it at, and mount it: mkdir -p /mnt/media mount /dev/sdb /mnt/media
    You'll see: nothing dramatic, which means success. Run df -h and your ~8T drive now shows against /mnt/media. It's part of the filesystem.
  2. Make it mount automatically at every boot. First get the drive's unique ID: blkid /dev/sdb Copy the UUID="..." value it prints. Open the config file in the built-in editor: nano /etc/fstab add this line at the bottom (paste your UUID in place of the x's), then save with Ctrl+O, Enter, Ctrl+X: UUID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx /mnt/media ext4 defaults 0 2
    What fstab is: a file listing drives Linux should mount automatically at startup. Without this line, the drive would disappear after a reboot and Jellyfin would lose its library. Editing config files like this by hand is a big part of real Linux administration, so this is good practice, not just a chore.

Step D, open a doorway into the Jellyfin container (bind mount)

Understand this: why the container can't see the drive yet Remember, a container is deliberately walled off from the host; that isolation is the point. So even though the host can now see /mnt/media, your Jellyfin container cannot, by design. You open one specific, controlled doorway called a bind mount: it makes a single host folder appear inside the container. You're not exposing the whole host, just sharing the one media folder in.
  1. Find your Jellyfin container's ID (the number next to it in Proxmox, e.g. 101), then from the host Shell attach the folder into it: pct set 101 -mp0 /mnt/media,mp=/media
    Reading it: pct is Proxmox's container tool. This says "for container 101, add mount point 0: share the host's /mnt/media so it appears inside as /media." Now your big drive is visible at /media from inside Jellyfin.
The genuinely fiddly bit: permissions One real gotcha. For security, "unprivileged" containers remap their internal user accounts to different numbers on the host, so the container can be blocked from reading the shared folder even though it's right there. The fix is a one-line ownership or mapping tweak, but which one depends on how your Jellyfin container was created (helper script or by hand). Rather than guess and have it silently not work, note where you are when you hit this and confirm how you made the container; the exact command is a 30-second job once that's known.

Step E, point Jellyfin at it

  1. In Jellyfin's web page, add or edit your media library and set its folder to /media (or a subfolder like /media/films). Copy some video files onto the drive and let Jellyfin scan.
    You'll see: your films and shows appear in Jellyfin, now served from the big drive with terabytes to grow into.
You'll meet this again Filesystems, mounting, /etc/fstab and the single-tree layout are core Linux fundamentals, straight out of CompTIA Linux+. The "OS disk plus data disks" split is how storage works on cloud VMs. And editing a config file like fstab by hand is the everyday reality of Linux and cloud administration, so this section punches above its weight for learning.
Recap
  • SSD holds the system; the big NAS drive holds bulk media. Different jobs, different drives.
  • A raw drive needs a filesystem (ext4) before it can store files; formatting writes that structure.
  • Linux mounts drives into one folder tree; fstab makes it happen automatically at boot.
  • A bind mount shares one host folder into an isolated container; unprivileged containers may need a permissions tweak.
Check yourself

Q1. Why use the SSD for the system but the hard drive for media?

The SSD is fast but small, ideal for the OS and apps doing many small reads/writes; the hard drive is big and cheap per terabyte, ideal for bulk media read in steady streams, which doesn't need SSD speed.

Q2. What does "formatting" a new drive actually do?

It writes a filesystem (like ext4) onto the raw disk: the structure that lets the system store and find files. Until then the drive can't hold anything.

Q3. The Jellyfin container can't see the drive even though the host can. Why, and what opens it up?

Containers are isolated from the host by design, so they can't see host drives automatically. A bind mount shares a specific host folder (here /mnt/media) into the container (as /media).

8.Jellyfin: watching your media (clients, servers, ports and transcoding)

You've built a container once, so this reinforces it, then teaches the client/server model and what "transcoding" really means. Nothing here affects the household.

  1. Create a second container the same way (or, since it's fiddlier to wire up the media drive and graphics, use the community-scripts.org Jellyfin helper and choose to enable hardware acceleration). Either way you'll get a Jellyfin address ending :8096.
    Why a helper is fine here: Jellyfin plus graphics passthrough is exactly the "fiddly, low-learning" bit where a script earns its keep. You already learned container-building by hand with Pi-hole earlier; nothing lost.
  2. Open the :8096 address, create your admin account, and point Jellyfin at your media drive: set the library folder to /media (the drive you prepared in the previous section).
  3. Install the free Jellyfin app on a TV/phone, enter the server address, sign in, press play.
Understand this: the client/server model, and transcoding Jellyfin shows you the pattern behind almost all software you use. The server (on your box, port 8096) holds the data and does the work; the client (the app on your TV) requests it. Your web browser and websites, your email app and mail servers, all work this way.

Transcoding is converting video from one format or quality to another on the fly, because your phone on 4G can't handle the same huge file your TV can. Doing that maths on the main processor (CPU) is slow and heavy. Quick Sync is a dedicated video-processing circuit built into the Intel chip's graphics that does it far more efficiently. Choosing to offload work to specialised hardware instead of the general CPU is a real architecture decision you'll see again (it's also why AI uses GPUs).
You'll meet this again Client/server, ports and services are foundational to everything, and "hardware acceleration / offloading" connects straight to why cloud providers rent GPU instances. When AZ-900 lists Azure's specialised compute options, you'll recognise the same idea you used for Jellyfin.
Recap
  • Software is mostly client/server: a server provides, a client consumes, over a port.
  • Transcoding reshapes video on the fly; Quick Sync offloads it to purpose-built hardware.
Check yourself

Q1. In the client/server model, which side is the Jellyfin app on your phone, and which side is on your OptiPlex?

The phone app is the client (requests and plays); the OptiPlex runs the server (stores the media and serves it).

Q2. Why is hardware transcoding (Quick Sync) better than making the CPU do it?

Quick Sync is dedicated video hardware that does the conversion far more efficiently, freeing the CPU and letting more people stream at once.

Nothing about your library, your accounts, or your files changes by adding it, it's a different front door onto the same building, not a new building.

9.Remote access: the whole of home networking, and security

Reaching your server from outside is where the whole of home networking finally clicks. We'll do it the way that actually sticks: get the picture in plain terms first, then put the proper names to each piece once it makes sense.

The picture, in plain terms

When you're out of the house, you want your phone to reach your stuff at home. Picture your setup as an apartment building on a street:

  • Your one physical machine is the building.
  • Jellyfin and Pi-hole are flats inside it. Here's the bit that trips people up: even though they share the one machine, each flat has its own address on the street (that's what a container is, its own little computer with its own number).
  • The base system that runs the whole box is the front desk.
  • A badge is a pass that lets a device into a private, invite-only tunnel. Only your phone and the front desk have one.

So when you're away: your phone (with its badge) travels down a private tunnel to the front desk. The front desk can knock on any flat's door because they're all in the same building, so it passes you through to Jellyfin or Pi-hole, even though those flats have no badge of their own. That "make the front desk pass me through to the flats" bit is the one step people find confusing, and it really is all it is.

But how does that private tunnel actually get made, without ever opening the front door? Through a switchboard (a matchmaker). Both your phone and your home keep a line dialled out to the switchboard at all times, and dialling out is always allowed. When you want in, the switchboard checks that both are on your guest list, introduces them, and a private tunnel forms straight between the two. Because both sides reached outward first, the main front door never has to open at all. That last point is the whole reason it is safe.

Why none of this is sitting on the public internet

In plain terms: your home is a house with a single front door to the street, and that door only opens outward. Your devices can start conversations going out, but the street can't wander in uninvited. Putting servers inside the house doesn't unlock that door.

The names: NAT and a stateful firewall Your whole home shares one public IP address from your provider; inside, each device has a private IP. The router does NAT (Network Address Translation), letting many private devices share that one public address for outbound traffic, and it runs a stateful firewall that only lets back in the replies to conversations your devices started. Anything inbound that nobody asked for is dropped. That's the "door only opens outward", said properly, and it's why your server isn't reachable from outside by default. Not luck, just how these work.
The one thing never to do (and its name) Online you'll be told to switch on port forwarding to reach your server. In plain terms, that props your front door open to the entire street. In proper terms, it exposes a service straight to the whole internet, where bots constantly scan for exactly that, and it widens your attack surface (a real security term). We never do it. Tailscale gets you in without ever opening the door.

Set it up good command-line practice

  1. Make a free account at tailscale.com (sign in with an existing Google/Microsoft login).
    Plain version: this creates the "club" that only your own badged devices are allowed to join.
  2. In the Proxmox Shell, install and start Tailscale on the host (the front desk) with its official commands: curl -fsSL https://tailscale.com/install.sh | sh tailscale up
    Reading the commands: same pattern as Pi-hole, curl fetches the official installer and pipes it into a shell to run. tailscale up links this machine to your account. It prints a link; open it on your laptop to approve. This is you giving the front desk its badge.
  3. Now make the front desk a doorway to the whole home (find your range from the host's IP: 192.168.1.50 gives 192.168.1.0/24, the first three numbers of your own host's address followed by .0/24): tailscale up --advertise-routes=192.168.1.0/24 --accept-dns=false then approve that route once in the admin page at login.tailscale.com.
    This is the "pass me through to the flats" step from the diagram. --advertise-routes=192.168.1.0/24 hands the front desk the whole home "street" to forward you onto, and the approval click is just you confirming you're happy for it to. That's all the scary-looking line does.
    If you see warnings about "forwarding disabled": Tailscale is telling you the machine's own underlying ability to pass traffic through to other devices is switched off by default, a separate, lower-level Linux setting from anything Tailscale itself controls, and a sensible default since most computers only ever handle traffic meant for themselves. Turn it on with: echo 'net.ipv4.ip_forward = 1' | tee -a /etc/sysctl.d/99-tailscale.conf echo 'net.ipv6.conf.all.forwarding = 1' | tee -a /etc/sysctl.d/99-tailscale.conf sysctl -p /etc/sysctl.d/99-tailscale.conf No need to redo the tailscale up command afterward, this fixes a separate permission underneath it. A second warning about UDP GRO on your bridge is just a performance tweak, not a correctness problem, safe to leave alone.
  4. Install the Tailscale app on your phone, sign into the same account, toggle it on. Turn wifi off (mobile data) and open Jellyfin at its home :8096 address.
    You'll see: it plays over mobile data, through the private tunnel, nothing exposed to the public. Lose a phone? Remove it in the admin page and it instantly loses all access.
  5. Try one of your memorable addresses instead, like nadrijellyfin.com, while still away from home on mobile data. This one will not load yet, and that is expected.
    Why the plain IP address works but the memorable name does not: a name like nadrijellyfin.com only means anything because Pi-hole is the one answering for it. Your phone only learns to ask Pi-hole for DNS answers because your home router's DHCP hands that instruction out automatically, and that only happens to devices actually connected to your home network. A phone out on mobile data, even through the Tailscale tunnel, never received that instruction, so it falls back to the regular public internet's DNS, which has never heard of nadrijellyfin.com and fails.
    The fix, done once in the Tailscale admin console: on a computer, go to login.tailscale.com and click DNS in the sidebar. Under Nameservers, add a Global Nameserver and enter Pi-hole's address (192.168.1.97). Switch on Override DNS servers underneath it, then save. This is Tailscale's own way of doing the same job your router's DHCP normally does, but for devices that are only ever reachable through the tunnel: it tells every device in your tailnet "for DNS questions, ask this address," the same instruction your router already gives phones at home. Afterward, on the phone, toggle Tailscale off then on again so it picks up the new setting, and try nadrijellyfin.com again on mobile data. It should now resolve exactly like it does at home.
You'll meet this again This one section covers NAT, firewalls, public/private IPs, VPNs, CIDR subnets and zero-trust. That's a serious chunk of Network+ and Security+, and the networking concepts in AZ-900/AZ-104 (VNets, Network Security Groups, VPN Gateway) are the same ideas at cloud scale. You'll walk into that material already fluent.
Recap
  • NAT + a stateful firewall are why your server isn't public by default; inbound uninvited traffic is dropped.
  • Port forwarding deliberately exposes a service and widens your attack surface; we avoid it.
  • A VPN (Tailscale/WireGuard) gives safe remote access via an invite-only encrypted overlay: zero-trust.
  • CIDR like /24 describes a block of addresses; the same notation runs cloud networks.
Check yourself

Q1. Your server has never had port forwarding set up. Explain to a mate why a hacker on the internet can't just connect to it.

The router uses NAT and a stateful firewall: it only lets in replies to connections your own devices started. Unsolicited inbound traffic from the internet is dropped, so the server isn't reachable from outside.

Q2. Why is a VPN like Tailscale safer than port forwarding for remote access?

Port forwarding exposes a service to the entire internet. A VPN creates an encrypted, invite-only tunnel that only your own devices can join, so nothing is publicly reachable.

Q3. Jellyfin and Pi-hole run on the same machine as the host, so why does the host still have to "forward" your phone to them?

Because each container has its own address on the network, and only the host has Tailscale. Your phone reaches the host through the tunnel, and the host passes it on to those containers by their own addresses. That forwarding is subnet routing.

Q4. What does /24 mean in 192.168.1.0/24?

CIDR notation for a block of 256 addresses (192.168.1.0–.255), where the first 24 bits are the fixed network portion.

10.Filling your library on its own: Gelato, a streaming engine, and no downloads

Up to now, getting a new film or show into Jellyfin has meant finding it, downloading the whole file, and storing it on your own disk. This section takes a completely different route to a library that keeps itself current: instead of downloading and keeping anything, your server shows a placeholder for every title you could want, thousands of them, and only pulls the actual video from a fast streaming source at the moment someone presses play. The payoff is an enormous library that fills itself with no downloading, no waiting overnight, and no half finished files, in exchange for leaning on an outside service to do the heavy lifting. That trade is a real one, so it is spelled out plainly below before anything gets set up.

Understand this: warehouse versus shop window The old download approach is a warehouse. Every title physically arrives and sits on your shelves, yours forever, playing even with the internet down, but slow to fill and hungry for disk space. This new approach is a shop window. Your library shows a card for every title, thousands of them, but nothing is actually in the back room until someone asks, at which point it is fetched and played in seconds. The technical name for those cards is stub files, tiny one or two kilobyte markers that stand in for a real video file. Because nothing is stored, nothing stalls the way that Rick and Morty download did, but the shop only works while its supplier is open.
Understand this: the three pieces that make it work Three parts do this between them, each with one clear job. A debrid service (this build uses TorBox) is the engine: a company that keeps a huge pile of already fetched content sitting on fast servers and hands it to you over an ordinary web connection, which is exactly why playback is instant and there is no seeding or stalling to babysit. AIOStreams is the adapter: it gathers dozens of separate streaming sources, tidies and de-duplicates them, and hands your server one clean feed instead of a mess of eighty. Gelato is the Jellyfin plugin that ties it together: it writes the stub cards into your library, and when someone hits play it asks AIOStreams for a stream, gets one served from your debrid engine, and pipes it through Jellyfin to the screen. Three small pieces stand in for the whole six app download stack this guide used to describe.
The honest trade off, before you commit This route rents rather than owns. The day the TorBox subscription lapses, or the source your feed points at goes down, your library becomes thousands of placeholder cards with nothing behind them, and there is no local copy to fall back on. The download route was slow and fiddly, but what it produced sat on your own disk and worked forever with no account attached to it. TorBox is a few dollars a month, so this is not an expensive habit, but it is a running one. Neither route is wrong; this build deliberately picks convenience and a library that fills itself, and keeps open the option of pinning a small handful of true favourites to real disk later so they survive an outage.
Check this first: your Jellyfin version Gelato needs Jellyfin 10.11 or newer to install at all. Before anything else, open Jellyfin, go to Dashboard, then About, and read the version number. If it is 10.11 or higher you are ready. If it is older, updating Jellyfin is the first job, and worth a snapshot of its container first out of the habit from the ops section. This one check decides everything downstream, so it comes before any account is created or any container is touched.

Docker is still how one of these pieces runs

Understand this: two of the three pieces need nothing installed on your server It is worth being clear about where each piece actually lives, because it is less work than the old stack, not more. TorBox is simply an account on someone else's servers, so there is nothing to install for it at all. Gelato installs inside Jellyfin itself, as a plugin, from Jellyfin's own Dashboard. The only piece that runs on your own machine is AIOStreams, and it runs as a Docker container. You already built a Docker container earlier in this build, so AIOStreams becomes one more small entry in a docker-compose.yml recipe file there, brought up with the same one command style you already know. Docker lets one lightweight Linux container run self contained app packages, called container images, each described in that single plain text recipe file, rather than each app needing its own full Proxmox container.

The build, in order

The exact commands, tokens and screens for each of these get filled into this section as we do them together, the same way every other part of this guide was written from the real steps rather than ahead of them. What follows is the shape of the build so the whole path is visible before the first click.

  1. Confirm Jellyfin is 10.11 or newer, updating it first if not. This is the gate for everything else, so it is settled before anything is bought or built.
  2. Create a TorBox account and get its API token. This is the paid engine, and its token is the single credential the rest of the setup hands around, so it is worth storing in your notes file the moment you have it, the same documentation habit as every password so far.
  3. Deploy AIOStreams as a container in your existing Docker box, then open its configuration page and give it two things: your TorBox token, so it knows which engine to resolve streams through, and at least one search source so it has a catalogue to show. It hands back a single web address called a manifest, which is the one link Gelato needs.
  4. Install the Gelato plugin in Jellyfin, by adding its plugin repository under Dashboard, Plugins, Repositories, installing Gelato from the catalogue, and restarting Jellyfin when it asks. Then paste the AIOStreams manifest into Gelato's settings so the plugin knows where to get its catalogue and streams.
  5. Give Gelato its own library folders, one for films and a separate one for shows, kept apart from any real media, since a single folder shared between the two is the most common thing that trips Gelato up. Point a new Jellyfin Movies library and a new Shows library at those folders so the stub cards appear in their own clean rows.
  6. Test one play, end to end: open Jellyfin, pick a title Gelato has filled in, and press play. A working setup starts the stream within a few seconds, served through your debrid engine, with nothing downloaded to disk. That first successful play is the proof the whole chain is wired correctly.
Only tear the old stack down once this one plays If you are moving here from the old download stack rather than starting fresh, resist removing it until a title has actually streamed through Gelato at least once. The moment to retire Sonarr, Radarr, Prowlarr, qBittorrent, gluetun and Jellyseerr is after the new route is proven, not before, so the household is never left with a library that neither downloads nor streams. Once Gelato plays cleanly, the old docker compose down on the arr stack is safe to run.
Recap
  • This route keeps almost nothing on disk. It fills the library with tiny stub file placeholders and fetches the real video on demand when someone presses play, the shop window rather than the warehouse.
  • Three pieces do it: TorBox (the paid engine that makes playback instant), AIOStreams (the adapter that gathers sources into one feed, run as a Docker container), and the Gelato plugin inside Jellyfin (which writes the stubs and resolves a stream on play).
  • The trade is convenience for ownership: it fills itself and never stalls, but it rents from an outside service, so nothing plays if that service or subscription goes away. Gelato needs Jellyfin 10.11 or newer.
Check yourself

Q1. Your library shows thousands of films but the disk is nearly empty. Where is the actual video for a film you have not watched yet?

Nowhere on your disk. Each title is only a tiny stub file placeholder until you press play, at which point Gelato fetches the real stream on demand from your debrid engine through AIOStreams. Nothing is stored unless you deliberately pin it later.

Q2. What is the one thing that stops working if your TorBox subscription lapses, and why is that different from the old download setup?

Playback stops entirely, because the debrid engine that actually serves the video is gone, leaving only empty placeholder cards. The old download setup kept real files on your own disk, so they would keep playing regardless of any subscription.

You'll meet this again The rent versus own decision you just made is the same one every organisation makes between streaming or a content delivery network and local storage, and between renting cloud capacity and buying your own hardware, an explicit cost and reliability trade in AZ-900's pricing and capex versus opex topics. Extending an application by dropping in a plugin, and wiring services together by handing one an API token for another, are both everyday patterns in real software integration. And Docker with a docker-compose.yml is still exactly the infrastructure as code idea behind Azure Container Instances and Kubernetes at cloud scale, describing what should run in one file rather than clicking through installs by hand.

11.One front door for the family: a dashboard, a memorable address, and no ports

Right now, reaching anything means knowing its exact IP address and port, which is fine for you but not realistic for the rest of the household. This section fixes that with two small pieces working together: a dashboard that turns every service into a single clickable tile, and one memorable address for the whole family to type instead of any of them.

Understand this: the two separate jobs being solved here A dashboard solves "where do I click", one page with an icon per service, so nobody needs to remember or even see an IP or port, they just tap Jellyfin and land in Jellyfin. A memorable local DNS name solves "what do I type to get there in the first place", one easy word instead of a string of numbers. Together they mean a family member only ever needs to know one thing.

Step A, the dashboard itself

Create it the same way as everything else, run from the Proxmox host:

bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/homarr.sh)"

Defaults: 2 cores, 2GB memory, 8GB disk, reached at <its-ip>:7575. Unlike most of the other containers, this one is entirely configured through its own web page rather than any config file, open it, and for each service, add a tile, give it a name and icon, and paste in its address (for example http://192.168.1.97/admin for Pi-hole). Drag them into whatever layout makes sense. Once it's up and working, set a DHCP reservation for it in your router, same habit as every other container.

Step B, one name for the whole house

Understand this: why Pi-hole is the right tool for this too Pi-hole is already the phone book the whole house uses to look up names. Adding your own entry to that phone book is exactly the same idea as it looking up bbc.co.uk, except this entry only exists inside your own house, nobody on the outside internet has ever heard of it, and it can point at whatever you like.
A genuine gotcha worth knowing before you pick names Browsers keep an internal list of endings they trust as real, .com, .org, and so on. Anything invented, like .home or .lan, isn't on that list, and modern browsers (Firefox especially) will quietly send an invented ending to a search engine instead of trying to visit it, purely as a safety precaution. The fix is simple: use a made-up name that still ends in a real, recognised TLD like .com. You never register or pay for it, Pi-hole simply answers for it entirely inside your own house before the request would ever reach the real internet, and because the ending looks legitimate, every browser trusts it and navigates straight there.

Worth picking a consistent pattern rather than a different style each time. This guide uses nadri<service>.com, so nadrihome.com for the dashboard, nadrijellyfin.com, nadripihole.com, and so on as you add more, swap in whatever prefix you like.

  1. In Pi-hole's admin page, go to Local DNS, then DNS Records.
  2. Add a new record: in the domain field enter nadrihome.com, and in the IP field enter just the bare IP address, Homarr's address on its own, nothing else typed alongside it. Click Add.
    Only the address goes in that box: no http://, no port number, no trailing slash. A DNS record can only ever answer "what address does this name mean", it has no way to carry a port or protocol alongside it, so anything extra typed in there gets rejected.
  3. On any device already using Pi-hole for DNS, open a browser and go to http://nadrihome.com:7575.
    You'll see: Homarr's dashboard loads. Notice the port number still had to be typed, that's the one piece a plain DNS name can never remove on its own, which is exactly what Step C below finishes off.

Step C, dropping the port number too: a reverse proxy

Understand this: why the port can't be avoided without one more piece When a browser gets a name with no port typed after it, it silently assumes port 80 (or 443 for a locked https:// address). Homarr doesn't live on port 80, it lives on 7575, so typing the name alone lands nowhere until something is actually sitting on port 80 ready to answer. A reverse proxy is that something: a single doorway on the normal port that looks at which name was asked for, and quietly forwards the visitor on to whichever real container and port actually handles it, Homarr, Jellyfin, Pi-hole, whatever it's told to. Every name can then work with nothing typed after it at all.

Create it the same way as everything else, run from the Proxmox host:

bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/nginxproxymanager.sh)"

Defaults: 2 cores, 2GB memory, 8GB disk. Its own admin page runs on port 81, reached at <its-ip>:81, that one port is the only exception you'll ever need to type, since it's how you manage the proxy itself, not one of the family's everyday addresses. On first visit it walks you straight into creating your own admin account, no default login to worry about.

  1. Inside Nginx Proxy Manager, go to Hosts, then Proxy Hosts, then Add Proxy Host.
  2. Under Domain Names, enter nadrihome.com. Under Forward Hostname / IP, enter Homarr's actual address. Under Forward Port, enter 7575. Save.
    What you just told it: "when someone asks for nadrihome.com, quietly walk them through to Homarr's real address on port 7575." The visitor never sees or types that port at all.
  3. Back in Pi-hole's Local DNS Records, edit the nadrihome.com record you made in Step B so its IP now points at the proxy's address instead of Homarr's own address directly.
    Why this changes: every name now needs to be pointed at the one doorway (the proxy), which then decides internally where each name actually goes. Pointing straight at Homarr again would skip the doorway and land you back needing the port.
  4. Repeat both steps, a Pi-hole record plus a matching Proxy Host entry, for each additional service, nadrijellyfin.com forwarding to Jellyfin's address on port 8096, nadripihole.com forwarding to Pi-hole's address on port 80, and so on, all sharing the one nadri<service>.com pattern.
  5. Test it: http://nadrihome.com, with nothing after it at all.
    You'll see: Homarr's dashboard, no port typed anywhere. That's the whole chain working end to end, a name resolves to the proxy, and the proxy quietly finishes the journey.

Step D, when a service lives one level deeper: Pi-hole's /admin

Understand this: not every service greets you at its own front door Some services put their actual interface one level in, at a specific path, rather than right at the address itself. Pi-hole is exactly this: visit its address alone and you land on a mostly blank holding page, the real dashboard lives at /admin on top of that address. Typing nadripihole.com/admin every time works, but it defeats the point of a short, memorable address, so the fix is to teach the proxy to automatically take that extra step for you.

In Nginx Proxy Manager, open the Proxy Host you made for Pi-hole (nadripihole.com), and click into its Advanced tab. Paste this into the custom configuration box:

location = / { return 301 /admin; }

Save.

Reading it: this says "if someone arrives at exactly the bare address (/), send back a note telling their browser it's moved to /admin." The 301 is the standard code meaning "moved permanently", and getting one is what makes the browser automatically ask again at the new location, without you typing anything further.
  1. Test it: http://nadripihole.com, nothing typed after it.
    You'll see: the address bar itself updates to show /admin added on the end, and Pi-hole's actual login page loads, all without you typing it yourself.

The same trick applies to any future service that hides its real interface behind its own path rather than its own root, just swap /admin for whatever that service's actual path turns out to be.

You'll meet this again This is a small taste of what internal DNS and service discovery do inside real organisations at a much larger scale, and a reverse proxy is one of the single most common pieces of real infrastructure anywhere, the exact same job Azure's Application Gateway or a load balancer does in front of cloud services, just at home-lab scale. The redirect trick above is a plain example of an HTTP status code doing real work, the three-digit numbers (301, 404, 200 and so on) that every web request and response carries, worth recognising them as a concept you'll meet constantly from here on.
Recap
  • A dashboard (Homarr) turns every service into one clickable tile, hiding IPs and ports from whoever's using it.
  • A local DNS record in Pi-hole gives the whole house one memorable name instead of an address. Ending it in a real, recognised TLD like .com keeps browsers from treating it as a search, even though it's never actually registered on the public internet. A DNS record can only ever hold a bare address, never a port or protocol.
  • A reverse proxy sits on the normal port and forwards each name on to its real container and port internally, which is what finally lets a name work with nothing typed after it.
  • A service that lives behind its own path, like Pi-hole's /admin, needs one extra rule in the proxy, a 301 redirect, so the bare address automatically continues on to the right place.
Check yourself

Q1. Why does a made-up name like nadrihome.home risk triggering a search instead of navigating, while nadrihome.com doesn't, even though neither is a real registered website?

Browsers only trust endings they recognise as real, registered TLDs, like .com. An invented ending like .home isn't on that list, so browsers assume it's probably a typo and search instead. Ending the made-up name in a real TLD keeps the browser's trust, even though Pi-hole is answering for it entirely locally and it was never actually registered.

Q2. Once the reverse proxy is in place, what does each Pi-hole DNS record need to point at, and why?

The proxy's own address, not each service's own address directly. The proxy is the one thing listening on the normal port and deciding where each name actually goes internally, so every name needs to be routed through it first.

Q3. Why does Pi-hole need an extra rule in the proxy that Homarr and Jellyfin don't?

Pi-hole's real dashboard lives one level in, at /admin, rather than at its own root address. Without a redirect rule, visiting the bare address lands on its mostly blank holding page instead of the login screen.

12.Ops discipline: the habits that separate hobbyists from pros

You now have a real, working system that the household relies on. These are the habits that keep it healthy, and they're the same disciplines that separate a hobbyist from someone doing this professionally. Build them in now, before you go adding more.

Understand this: a snapshot is not a backup A snapshot saves a container's state on the same disk: brilliant for "undo my change", useless if the disk itself dies. A backup is a separate copy kept elsewhere. Pros follow the 3-2-1 rule: 3 copies of important data, on 2 different media, with 1 kept off-site. Even a mirror of two drives isn't a backup, because deleting a file deletes it on both. This distinction is a genuine exam and interview topic.
  • Snapshot before every change (container → Snapshots → Take Snapshot). Ten-second undo.
  • Document everything in your notes file: IPs, ports, passwords, what each container does. Ops teams live and die by good documentation.
  • Update monthly: snapshot, then apt update && apt upgrade, then check things still work. Patching is basic security hygiene.
  • Least privilege: don't run things as root when you don't need to, and prefer "unprivileged" containers. Giving software only the access it needs is a core security principle.
  • Nothing is exposed unless you deliberately port-forward, which we never do.
Recap
  • Snapshots undo changes; backups (3-2-1) survive disasters. They're different.
  • Documentation, regular patching, and least privilege are the everyday disciplines of real operations and security work.

13.The roadmap: the other apps worth adding

The rest of what was floated early on, each its own separate container, none set up yet. Every one of these follows the exact pattern you already know by heart: run the install command from the Proxmox host, accept the defaults, open the address it gives you, set a reserved address in your router once it's working. What's below is enough to actually start each one, not just a name and a sentence.

Immich, family photo backup

What it doesEvery phone in the house backs up its photos here automatically, in the background, the same everyday behaviour as Google Photos or iCloud, except the actual copies live on your own NAS drive rather than a company's servers.
bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/immich.sh)"

Defaults: 4 cores, 6GB memory, 20GB disk, reached at <its-ip>:2283. One thing worth knowing before you start it: the very first install compiles several components from scratch, which can take anywhere from about 15 minutes to a couple of hours depending on how busy the machine is, that's normal and not a hang. Once it's up, create an admin account on first visit, then install the Immich app on each family member's phone and point it at that address to start backing up automatically. Its own photo storage lives on the container's small disk by default, so before everyone starts uploading in earnest, this is one worth bind mounting to your NAS drive the same way Jellyfin's media folder was, so photos land on the big drive rather than filling up a 20GB container disk.

Nextcloud, a shared family drive

What it doesA shared space for documents and files the whole family can drop things into and get to from any device, your own version of Google Drive or Dropbox.
bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/nextcloudpi.sh)"

Defaults: 2 cores, 2GB memory, 8GB disk, reached at https://<its-ip> (it uses HTTPS on port 443 by default, so expect the same self-signed certificate warning you saw with Proxmox itself, safe to click through on your own network). One honest note directly from the project's own documentation: this particular installer pulls its code from outside the main community-scripts repository, so it's worth a quick look at what it's fetching before running it, same "only run things from a source you trust" habit from earlier in this guide. Same as Immich, plan to bind mount your NAS drive in for the actual file storage rather than relying on the small default disk once real use begins.

Vaultwarden, a shared password manager

What it doesOne place for the whole family to store logins and passwords securely, using the same free apps as the popular password manager Bitwarden, just pointed at your own server instead of their company's.
bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/vaultwarden.sh)"

Defaults: 4 cores, 6GB memory, 20GB disk, reached at <its-ip>:8000. On first visit, create your own account, then each family member installs the free Bitwarden app and, instead of using Bitwarden's own servers, points it at your server's address in its settings, then creates their own login there. Nothing this container holds needs to touch your NAS drive, its own small disk is plenty.

Home Assistant, smart home control

What it doesOne dashboard for controlling smart plugs, lights, sensors and similar devices, instead of a different app per brand.
bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/homeassistant.sh)"

Defaults: 2 cores, 2GB memory, 16GB disk, reached at <its-ip>:8123. Worth knowing before you pick this one: this particular install is the leaner "Core" version, meaning it does not support the wider world of optional community add-ons the full Home Assistant OS offers, a reasonable trade for how much simpler it is to run inside a container. Follow its own setup wizard on first visit to create an account and start adding devices.

Uptime Kuma, a dashboard that watches everything else

What it doesA small dashboard that quietly checks whether your other containers are actually up and responding, and tells you the moment one isn't, so you find out from a notification rather than a family member complaining Jellyfin's not working.
bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/uptimekuma.sh)"

Defaults: 1 core, 1GB memory, 4GB disk, reached at <its-ip>:3001. On first visit, create an account, then add a "monitor" for each container's address (Pi-hole, Jellyfin, and so on), it'll check on them automatically on a schedule and can be set to email or notify you the moment one stops responding.

Proxmox Backup Server, the real off-site backup

What it does, and why it's different from a snapshotThis is the actual "1 copy off-site" piece of the 3-2-1 rule from the ops discipline section. A snapshot lives on the same disk as everything else, so it's no help at all if that disk fails. Proxmox Backup Server keeps proper, separate, restorable backups of your containers.
bash -c "$(curl -fsSL https://raw.githubusercontent.com/community-scripts/ProxmoxVE/main/ct/proxmox-backup-server.sh)"

Defaults: 2 cores, 2GB memory, 10GB disk, reached at <its-ip>:8007. Being straightforward about a genuine limitation here: running this as a container on the exact same physical machine as everything it's backing up protects you against a corrupted container or a mistake, but not against the machine itself dying entirely, drive failure, theft, fire, since the backups would go down with it. The properly correct setup, whenever you're ready for it, is this same software running on a second, separate physical machine, ideally not even in the same room. Starting it here on the same OptiPlex is still a genuine improvement over snapshots alone, just worth knowing what it doesn't protect against yet.

Every one of these follows the same pattern you already know by heart now: a container, a bind mount if it needs to touch shared storage, a reserved address, and its own login. Nothing about how they get added is fundamentally new territory anymore, that's the real payoff of everything covered so far. Pick them off one at a time whenever you fancy, in any order, snapshotting first out of the habit from the last section.

Recap
  • Immich, Nextcloud, Vaultwarden, Home Assistant, Uptime Kuma and Proxmox Backup Server are all optional next containers, each doing one clear job.
  • They all install and run the same way as everything you've already built, so none of them is new territory.
  • Proxmox Backup Server only becomes a true off-site backup once it lives on a second, separate machine.

15.Glossary of every term you'll meet

Fladder
An alternative app for watching your Jellyfin library, built by a different team. Talks to the same server, no changes needed on Jellyfin's side.
Homarr
A dashboard container that shows every one of your services as a clickable tile, so nobody needs to know an IP address or port to reach anything.
Local DNS record
A name-to-address entry you add to Pi-hole yourself, only known inside your own house, letting a memorable name like nadrihome.com replace an IP address. Ending it in a real TLD like .com keeps browsers from treating it as a search. Can only ever hold a bare address, never a port or protocol.
Reverse proxy
A container (here, Nginx Proxy Manager) that sits on the normal web port and forwards each incoming name on to the right internal container and port, which is what lets a name work with no port typed after it.
HTTP status code / 301 redirect
A three-digit number every web response carries, telling the browser what happened. 301 means "moved permanently", telling the browser to automatically continue on to a new address, used here to send a bare address on to a service's real path like /admin.
Gelato
A Jellyfin plugin that fills your library with placeholder cards for thousands of titles and, when you press play, fetches a real stream on demand rather than storing anything. Needs Jellyfin 10.11 or newer.
AIOStreams
An adapter that gathers many separate streaming sources into one tidy, de-duplicated feed for Gelato to read from. The one piece of this route that runs as a container on your own machine.
Debrid / TorBox
A paid service that keeps a large pile of already fetched content on fast servers and serves it over an ordinary web connection, which is what makes on-demand playback instant with no seeding or stalling. TorBox is the one this build uses.
Stub file
A tiny one or two kilobyte placeholder standing in for a real media file. Gelato fills the library with these so it looks complete while almost nothing is actually stored on disk.
Manifest
The single web address AIOStreams produces once configured, which you paste into Gelato so the plugin knows where to pull its catalogue and streams from.
Docker / Docker Compose
Docker runs small, lightweight app packages (container images) inside one Linux container, sharing its resources rather than each getting a full separate one. Docker Compose is a single recipe file (docker-compose.yml) listing every app to run, what to give it access to, and how they connect, applied in one command.
Immich / Nextcloud / Vaultwarden / Home Assistant / Uptime Kuma / Proxmox Backup Server
Future containers on the roadmap: private photo backup, a shared drive, a shared password manager, smart home control, a dashboard that watches your other containers, and a proper off-site backup system.
Hypervisor
Software that runs multiple virtual computers on one physical machine. Proxmox is a Type 1 (bare-metal) one.
Virtual machine (VM)
A software "computer" that runs its own full operating system on a hypervisor.
Container / LXC
A lighter isolated environment that shares the host's Linux kernel. Docker and Kubernetes are the industry versions.
Template / image
A pre-built base OS a container or VM is created from.
Linux
The open-source operating system running most servers and clouds. Debian is the flavour under Proxmox.
Shell / terminal / CLI
The text command-line interface for controlling a computer by typing.
root
The all-powerful administrator account on Linux.
Package manager / apt / repository
The tool (apt) that installs software from curated online libraries (repositories), handling dependencies.
IP address
A device's network number. Private ones (192.168.x, 10.x, 172.16-31.x) only work inside your network.
DNS
The internet's phone book: turns names into IP addresses. Runs on port 53.
DHCP
The service that automatically hands out IP addresses to devices joining a network.
Port
A number identifying a specific service on a machine (Proxmox 8006, DNS 53, Jellyfin 8096).
NAT
Network Address Translation: lets many private devices share one public IP, and blocks uninvited inbound traffic.
Firewall
Filters network traffic by rules. A stateful one only allows replies to connections you started.
Port forwarding
Deliberately opening a service to the internet through the firewall. Risky; we avoid it.
VPN / WireGuard / Tailscale
An encrypted private network over the internet for your own devices. Tailscale uses the WireGuard protocol.
CIDR (e.g. /24)
Notation for a block of IP addresses; /24 = 256 addresses. Used to define cloud networks too.
Zero-trust
Security approach: trust no network by default, verify every device. Tailscale is an example.
IP forwarding
A Linux setting, separate from Tailscale itself, controlling whether a machine is allowed to pass traffic through to other devices rather than only handling traffic meant for itself. Off by default; needs enabling for subnet routing to actually work.
Global Nameserver (Tailscale DNS)
A DNS address that Tailscale hands out to every device in the tailnet, the same role a home router's DHCP normally plays for devices connected to it locally. Needed so devices that are only ever reachable through the tunnel (like a phone away from home) still know to ask Pi-hole for custom domain names instead of falling back to the public internet's DNS.
Client / server
The server provides a service; the client consumes it. The pattern behind most software.
Transcoding / Quick Sync
Reshaping video on the fly; Quick Sync is Intel hardware that does it efficiently.
Snapshot vs backup
A snapshot is an on-disk undo point; a backup is a separate copy elsewhere (see 3-2-1).
Headless
A machine run with no monitor/keyboard, managed remotely over the network.
IaaS
Infrastructure as a Service: renting virtual machines/networks from a cloud. An AZ-900 topic you've now built by hand.
